Grounding Context

Codebase Context. The repository’s E5 commitment-surface experiment found that correctly labeled deployment-support coverage beat hoped-for generator learning: the coverage arm reached canonical OOD 0.741 while generator regularization reached 0.063. Gauge-fixed concern work argues that observationally equivalent descriptions need separating anchors, and that claimed structure matters only when it changes the deployed commitment surface. A registered semantic concern experiment also produced wrong-sign transfer, warning against making latent similarity the sole judge.

External work supports structure-first UI decomposition, executable render scoring, component-to-code mappings, and localized repair. It does not yet establish that any one proposed integration below will beat a strong screenshot-plus-browser baseline. Confidence therefore reflects mechanism evidence and implementation risk separately.

1Deployment Capsule Library

Description: Build landing-page primitives as deployment capsules. Each capsule contains a real production component, prop schema, tokens, and labeled browser renders for desktop, mobile, long copy, missing images, hover, and focus. GPT Image 2 is asked to art-direct and compose from these known primitives, while clearly marked custom zones remain free-form. The coding model receives capsule IDs and composition parameters rather than recreating known UI from pixels.

Browser-proven capsules→GPT Image 2 composition→Assembly, not reconstruction
Axis
Known components and deployment coverage
Basis
direct: Repository E5 found correct deployment-support coverage competitive with the structured intervention while generator regularization failed. external: Figma Code Connect demonstrates mapping designed components and variants to production imports and props.
Rationale
This moves coverage upstream into the design process. The screenshot is less ambiguous because most visible structures already have one canonical implementation and verified responsive states.
Downsides
Up-front atlas maintenance, design lock-in, and a combinatorial state surface. The claim that GPT Image 2 will obey capsule IDs must be mediated by a typed composition manifest outside the pixels.

Complexity
Medium
Best practical bet. Start with 8–12 section families and test whether at least 70% of the current landing page can be assembled without visibly weakening the concept.

2Executable Design Genome

Description: Make a typed, renderable design intermediate representation the source of truth: section tree, anchors, responsive constraints, text-wrap targets, asset IDs, tokens, component states, and explicit uncertainty. A deterministic compiler produces the initial scaffold and crude reference previews. The coding model acts as a backend for bounded custom components and integration instead of owning the full inverse problem from pixels.

Chosen visual direction→Validated layout genome→Deterministic scaffold + code
Axis
Structured representation
Basis
direct: Repository work favors typed artifact contracts, explicit gauge-fixing anchors, and richer representations when identification stalls. external: UICopilot and DCGen support structure-first decomposition and region-wise synthesis, though not this full compiler claim.
Rationale
Most costly corrections repair unspecified decisions: crop versus scale, fixed versus fluid spacing, intentional wrapping, and breakpoint transformations. A validated IR can fail those decisions before implementation.
Downsides
The hardest step—faithfully extracting or authoring responsive constraints—remains unproved. A large schema or solver can create false precision, burden novel layouts, and cost more than it saves.
Confidence
61%
Complexity
High
Research bet. Encode only the current nav and hero in a minimal schema, compile both viewports, and stop if equal-budget fidelity does not materially beat ordinary screenshot handoff.

3Section-Champion Portfolio

Description: Generate three structurally diverse implementations in parallel, render each under frozen browser conditions, and score them section by section. Hard gates reject build, copy, asset, overflow, and interaction failures; separate metrics evaluate section coverage, geometry, typography, color, and responsive behavior. A final composition uses the champion hero, proof block, CTA, and other sections, followed by one selector-addressed integration correction.

3 diverse implementations→Section-level browser arbitration→Champion composition
Axis
Parallel selection and verification
Basis
external: Design2Code, executable render-and-filter work, Playwright screenshot comparison, and localized visual repair support the ingredients. reasoned: section selection preserves correct local work that whole-page best-of-N discards.
Rationale
Serial repair inherits the first implementation’s structural mistakes. Parallel structural hypotheses make decomposition selectable without adding conversational depth.
Downsides
Three to four times the generation cost. Cross-candidate sections may disagree on page rhythm, CSS assumptions, accessibility, or breakpoints; the promised one-pass seam repair is not established by prior work.
Confidence
54%
Complexity
Medium–High
Freshest but least proven. Pilot only the hero and social-proof sections. Measure whether section-level oracle selection beats the best complete candidate after accounting for seam corrections.
